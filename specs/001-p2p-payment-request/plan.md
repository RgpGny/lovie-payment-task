# Implementation Plan: P2P Payment Request

**Branch**: `001-p2p-payment-request` | **Date**: 2026-04-17 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-p2p-payment-request/spec.md`

## Summary

Ship a Venmo-style P2P payment request feature: authenticated users create money requests addressed to a recipient email, recipients pay or decline, senders can cancel, requests expire after 7 days, and each request has a unique shareable public link. The technical approach is a Next.js 16 App Router application backed by Supabase Postgres. Row Level Security is the primary authorization boundary; money is stored as integer cents; all terminal status transitions use an atomic `UPDATE ... WHERE status='pending' AND expires_at > now()` guard; expiration is computed at read time (no background jobs). Zod schemas are shared between the browser form and the server API for identical validation on both sides.

## Technical Context

**Language/Version**: TypeScript 5 (strict mode), targeting Node.js 24 LTS via Vercel Fluid Compute.
**Primary Dependencies**: Next.js 16, @supabase/ssr + @supabase/supabase-js, Zod, nanoid, Tailwind CSS v4, shadcn/ui (button, card, input, label, badge, tabs, sonner, dropdown-menu), lucide-react, sonner, class-variance-authority, clsx, tailwind-merge.
**Storage**: Supabase Postgres (project `omxnewprtonfxtalzmco`). Two tables вЂ” `profiles` (1-to-1 with `auth.users`) and `payment_requests` вЂ” plus one view `public_requests` granted to `anon` for share-link reads. `request_status` enum with values `pending | paid | declined | cancelled`.
**Testing**: `@playwright/test` for E2E with `video: 'on'`, single chromium project. No unit-test suite (out of scope for a one-day ship; Zod/TypeScript cover most unit-level correctness).
**Target Platform**: Web вЂ” latest two versions of Chrome, Firefox, Safari. Deployed to Vercel Fluid Compute (Node.js runtime). Supabase region follows project default.
**Project Type**: Web application вЂ” single Next.js app with co-located server (API routes) and client (React Server Components + Client Components).
**Performance Goals**: Dashboard list under 1s at p95 for accounts with в‰¤100 requests (SC-007). Share-page render under 1s at p95 (SC-006). Create/Pay/Decline/Cancel actions under 500ms server-side.
**Constraints**: Money is integer cents at every layer (input в†’ validation в†’ DB в†’ response в†’ display). No background workers or cron jobs вЂ” expiration is a read-time predicate. All status transitions are atomic via a single guarded UPDATE. RLS is the ONLY authorization layer for non-public reads; middleware is a UX redirect, not an auth boundary.
**Scale/Scope**: Interview-assignment shape вЂ” tens of users, tens of requests per user. Single region. No audit log, no admin panel.

## Phase 0 вЂ” Research Summary

See `research.md` for the full write-up. Decisions taken:

- **Auth** в†’ Supabase email + password; sessions managed via `@supabase/ssr` cookies.
- **Authorization** в†’ Row Level Security on `payment_requests`. Middleware is a UX guard only.
- **Concurrency** в†’ Atomic `UPDATE ... WHERE status='pending' AND expires_at > now()` for every terminal transition. The `UPDATE` either affects 1 row (success) or 0 rows (race lost or expired). API returns 409 on 0 rows.
- **Expiration** в†’ Stored `expires_at = created_at + interval '7 days'`. Read-time `now() >= expires_at` predicate. No cron.
- **Share links** в†’ `nanoid(21)` token stored in `share_link`. A `public_requests` view is granted to `anon` so unauthenticated users can read via the token without touching the raw table.
- **Money** в†’ `amount_cents integer` with a `check (amount_cents > 0 and amount_cents <= 99999999)` domain constraint. Display-only conversion via `Intl.NumberFormat`.
- **Validation** в†’ Zod schemas in `src/lib/validators.ts`, imported by the form (`"use client"`) and by server route handlers. Server call is the trust boundary.
- **Testing** в†’ Playwright only; `video: 'on'` because the interview rubric values demonstrable coverage.
